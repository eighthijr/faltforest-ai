# QA Review Komprehensif — UI, UX, dan Bisnis

Tanggal review: 26 Maret 2026  
Reviewer role: QA Expert

## Ringkasan Eksekutif

Secara umum, codebase sudah menunjukkan fondasi produk SaaS yang jelas: funnel dari landing → auth → dashboard → workspace → pricing/payment sudah terhubung dan ada pemisahan layer komponen vs service API yang cukup rapi. Namun, dari perspektif kualitas produk, masih ada gap penting yang berpengaruh langsung ke conversion, trust user, dan reliability operasional.

Status keseluruhan saat ini:

- **UI maturity: 6/10** (sudah usable, tetapi konsistensi visual belum solid)
- **UX maturity: 5.5/10** (alur utama jalan, namun masih ada friction di onboarding, accessibility, dan feedback state)
- **Business readiness: 6/10** (model monetisasi sudah ada, tetapi observability funnel, trust cue, dan reliability gateway masih berisiko)

## Metodologi Review

Review dilakukan lewat:

1. Audit struktur flow utama (landing, auth, dashboard, workspace, pricing/payment).
2. Audit implementasi komponen UI utama (state handling, empty/error/loading state, CTA clarity).
3. Audit implikasi bisnis (paywall logic, payment behavior, analytics coverage).
4. Penilaian risiko berdasarkan dampak ke conversion, retention, dan revenue leakage.

## 1) Review UI (Visual & Interface Quality)

### Kekuatan

- Struktur halaman utama cukup jelas dan modular (`LandingPage`, `ProjectDashboard`, `WorkspaceChat`).
- Komponen status visual dasar sudah ada: toast, spinner, badge package FREE/PREMIUM, progress bar brief.
- Visual hierarchy di dashboard/workspace cukup terbaca (header → body → action).

### Temuan UI yang Perlu Diperbaiki

#### 1.1 Inkonistensi desain lintas halaman

- Landing + Auth menggunakan CSS custom (`landing-page.css`, `auth-modal.css`) sementara area lain banyak memakai Tailwind utility classes.
- Akibatnya ada potensi **inkonsistensi spacing, radius, warna, dan typographic rhythm** seiring scale produk.

**Dampak:** pengalaman visual terasa “campur” dan menurunkan persepsi kualitas brand premium.

#### 1.2 Komponen tabel dashboard belum mobile-friendly

- Dashboard project menampilkan tabel full column tanpa indikasi responsive behavior (misalnya horizontal scroll wrapper yang eksplisit untuk small viewport).
- Pada layar sempit, kolom `Project ID`, `Status`, `Dibuat`, `Aksi` berisiko padat/overflow.

**Dampak:** usability mobile menurun; user sulit melanjutkan ke workspace dari perangkat ponsel.

#### 1.3 Visual trust cue di pricing masih minimal

- Halaman pricing sudah jelas menampilkan paket, namun minim elemen trust visual seperti SLA, badge keamanan transaksi, estimasi verifikasi manual, dan “apa yang terjadi setelah bayar”.

**Dampak:** conversion upgrade bisa tertahan, terutama user baru yang belum percaya proses pembayaran.

## 2) Review UX (Flow, Clarity, Feedback, Accessibility)

### Kekuatan

- Alur inti onboarding cukup eksplisit: “Buat project → Workspace → Generate”.
- Error handling dan toast sudah diterapkan di banyak titik kritis.
- Workspace menyimpan snapshot localStorage sehingga sesi user relatif tahan refresh.

### Temuan UX Prioritas Tinggi

#### 2.1 Bahasa antarmuka campur ID + EN

- Contoh: modal auth menggunakan teks Inggris (“Continue with Google”, “Create account”), sedangkan mayoritas produk berbahasa Indonesia.

**Dampak:** tone produk terasa tidak konsisten dan bisa menambah cognitive load user non-teknis.

#### 2.2 Aksesibilitas keyboard/focus pada modal belum kuat

- Modal auth/paywall belum terlihat memiliki focus trap + escape handling + lock scroll body yang tegas.
- Close interaction ada tombol “x”, tetapi flow keyboard/screen reader belum lengkap.

**Dampak:** pengalaman pengguna keyboard/screen-reader kurang baik, berisiko compliance accessibility.

#### 2.3 UX failure state di payment belum sepenuhnya membimbing user

- Saat checkout/pembayaran gagal, feedback error ada, tetapi belum selalu memberi next-step terstruktur (retry path, fallback method, ETA konfirmasi manual, kontak support).

**Dampak:** user drop-off di tahap paling sensitif (money step).

#### 2.4 Workspace chat belum memberi validasi konteks jawaban

- User bisa mengisi jawaban sangat pendek/ambigu tanpa guardrail kualitas input.

**Dampak:** output generasi bisa rendah kualitas; user menyalahkan produk AI padahal brief minim.

## 3) Review Bisnis (Monetisasi, Risiko Revenue, Operasional)

### Kekuatan

- Segmentasi FREE vs PREMIUM sudah jelas di dashboard/workspace/pricing.
- Paywall enforcement sudah ditanam di server policy layer.
- Payment mendukung dua jalur: Tripay + manual proof.

### Temuan Bisnis Kritis

#### 3.1 Integrasi Tripay masih hardcoded ke sandbox endpoint

- Service payment mengarah ke `https://tripay.co.id/api-sandbox/transaction/create`.

**Dampak:** jika tidak diparameterisasi environment, ada risiko salah konfigurasi saat production dan menyebabkan failure transaksi nyata.

#### 3.2 Data customer pada request Tripay bersifat dummy

- `customer_name` dan `customer_email` masih statis (`User`, `user@example.com`).

**Dampak:** menurunkan kualitas audit trail transaksi, potensi mismatch rekonsiliasi, dan trust issue ketika dispute.

#### 3.3 Analytics funnel belum menangkap event bisnis granular

- Event utama sudah ada (page_view, cta_click, login_success, dll), tapi belum terlihat event granular khusus payment funnel (checkout_created, checkout_opened, proof_uploaded, proof_rejected, retry_success, dll).

**Dampak:** sulit mendiagnosis bottleneck conversion upgrade dan menghitung recovery opportunity.

#### 3.4 Polling status premium di pricing berpotensi boros dan noisy

- Polling setiap 5 detik berjalan selama belum success; ini aman sebagai baseline, tetapi belum ada strategi backoff/timeout UX.

**Dampak:** beban request meningkat saat traffic besar, user menunggu tanpa kepastian yang cukup.

## 4) Daftar Risiko Prioritas (Severity Matrix)

### P0 (kritikal untuk revenue/trust)

1. **Endpoint payment sandbox hardcoded**.
2. **Data customer payment dummy**.

### P1 (tinggi, memengaruhi conversion dan experience)

3. Inkonistensi bahasa UI.
4. Accessibility modal (focus trap/escape/scroll lock).
5. UX error recovery payment belum terstruktur.

### P2 (menengah, meningkatkan kualitas dan efisiensi)

6. Konsolidasi design system (token + komponen shared).
7. Responsive refinement tabel dashboard.
8. Granular analytics event untuk funnel pembayaran.

---

## 5) Rekomendasi Implementasi Bertahap

## Sprint 1 (Quick wins, 3–5 hari)

- Satukan bahasa antarmuka ke Bahasa Indonesia untuk flow utama conversion.
- Tambah “next action” pada semua error payment (coba lagi, ganti metode, tunggu verifikasi, hubungi support).
- Tambah microcopy trust di pricing: “verifikasi manual maksimal X jam”, “status bisa dilihat di halaman transaksi”, dan “upgrade aktif otomatis setelah sukses”.
- Tambah responsive handling dashboard table (stack card atau horizontal scroll yang jelas + sticky action).

## Sprint 2 (Reliability & conversion uplift, 1–2 minggu)

- Parameterisasi endpoint Tripay berdasarkan environment (sandbox/production) via env var.
- Kirim metadata customer riil (minimal user id + email masked) ke layer payment gateway.
- Tambah event analytics granular untuk payment funnel dan retry journey.
- Implement modal accessibility package: focus trap, ESC close, restore focus origin.

## Sprint 3 (Product excellence, 2–4 minggu)

- Bentuk mini design system internal (color token, spacing scale, button/input/modal variants).
- Validasi kualitas input brief di workspace (hint dinamis, contoh jawaban bagus, minimum quality checks).
- Tambah dashboard metrik bisnis operasional: conversion by step, payment fail reasons, approval SLA manual.

## 7) Kesimpulan

Produk ini sudah punya kerangka end-to-end yang baik dan bisa dipakai, terutama untuk MVP komersial. Fokus peningkatan berikutnya sebaiknya pada:

1. **Trust dan reliabilitas payment**,
2. **Konsistensi UX (bahasa + accessibility + recovery state)**,
3. **Ketajaman analytics funnel** agar keputusan bisnis tidak berbasis asumsi.

Jika 8 prioritas di atas dieksekusi, dampaknya paling cepat akan terlihat pada conversion upgrade dan penurunan drop-off pada tahap pembayaran.
